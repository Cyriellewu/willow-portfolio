Oracle OS: code-generated symbolic engines with constrained LLM interpretation.

A full-stack reflection workspace built on one implementation rule: rule-based application logic produces the symbols; a language model may interpret them, but does not generate a cast or make a prediction.

The question

Define a clear boundary between code-generated symbols and model output.

The product keeps symbol generation in code and limits the LLM to interpretation. This makes the system boundary visible to the user and avoids presenting model output as a deterministic forecast.

Local mock profile. The I Ching, BaZi, and Astrology lenses are generated through application logic.

The boundary

Rule-based engines first; model interpretation second.

I implemented four lenses - Tarot, I Ching, BaZi, and Astrology - plus Council mode for a multi-lens synthesis. The engines generate the symbolic material through application logic; LLM calls are isolated to reflect, chat, and council endpoints.

The loop

Journal motifs personalize subsequent prompts.

The system connects a daily cast to a journal, extracts recurring motifs, and uses those motifs to shape the next prompt. A code-only daily food draw and mock LLM fallback keep the demo useful even without API keys.

Code casts→ LLM reflects→ Journal motifs→ Tomorrow's prompt

Safety

Implemented constraints redirect high-risk prompt categories.

Medical, legal, and financial prompts are redirected. These implemented constraints sit alongside externalized prompts and API validation; they are product-level guardrails, not a claim of comprehensive safety.
