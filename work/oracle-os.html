Oracle OS draws a line between software, symbolism, and prediction.

A symbolic-reflection workspace built on one product rule: code generates the symbols; a language model may interpret them, but never invents the cast or presents it as deterministic fate.

The question

What should an AI reflection product be allowed to claim?

Many fortune-style products blur a random language-model response with a reading. I designed Oracle OS around a calmer premise: symbolic forecast, not deterministic fate.

Local mock profile. The I Ching, BaZi, and Astrology lenses are generated from deterministic application logic.

The boundary

Symbols come from code. Interpretation comes later.

I implemented four lenses - Tarot, I Ching, BaZi, and Astrology - plus Council mode for a multi-lens synthesis. The engines generate the symbolic material deterministically; LLM calls are isolated to reflect, chat, and council endpoints.

Deterministic engineCast symbols from code, never from an LLM
Interpretation layerLLM adds reflective language, not a claim of certainty
User contextJournal and motifs personalize future prompts

The loop

A reflection becomes useful only when it can return to a life.

The system connects a daily cast to a journal, extracts recurring motifs, and uses those motifs to shape the next prompt. A code-only daily food draw and mock LLM fallback keep the demo useful even without API keys.

Safety

Some questions should be redirected, not answered more confidently.

Medical, legal, and financial prompts are redirected. This is part of the product architecture, alongside externalized prompts and API validation - not a disclaimer pasted on after the model response.
