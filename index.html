How I tend to work

Start with the thing that does not yet make sense.

01

Reframe the question

"Predict failure" and "make a reflection app" are not requirements. I look for the decision, constraint, or person hiding behind the request.

02

Make the boundary visible

I prefer systems with explicit assumptions: what deterministic code owns, what a model can suggest, and where a result should stop being trusted.

03

Leave an evidence trail

A model metric, a test suite, an error trace, or a user-facing interface should make it possible to inspect what changed and why.
